# 第 4-6 节阅读笔记

## 1. 第 4 节：Dijkstra 的重新理解

* Dijkstra 不只是算最短路距离，它同时做三件事：
  * 算真实距离；
  * 找最短路树；
  * 给出一个 true distance order。

* 这篇论文真正关心的是第三件事，也就是顶点被扫描出来的顺序。

* Lemma 4.1 的意思：Dijkstra 扫描顶点的顺序就是一个真实距离顺序。

## 2. 堆实现和 `F - n + 1`

* 用堆实现 Dijkstra 时，堆里放的是 labeled 但还没有 scanned 的顶点。
* key 是当前距离。
* 每次扫描时用 `delete-min` 取出当前距离最小的顶点。
* 新发现顶点时用 `insert`。
* 已发现顶点距离变小时用 `decrease-key`。

Lemma 4.2 里比较重要的是：

* 堆外比较次数最多是 `F - n + 1`。
* `decrease-key` 次数最多也是 `F - n + 1`。
* `F` 不是孤立定义，它会控制 Dijkstra 中可能发生多少次额外比较。
* `n - 1` 条前向弧负责第一次发现顶点。
* 剩下的前向弧才可能导致比较或者 `decrease-key`。

## 3. 第 5 节：两个复杂度模型

### comparison model

* 算法知道图结构。
* 弧长通过 oracle 访问。
* 代价主要是比较弧长线性函数。
* 这个模型比较抽象，也比较强。

### time model

* 图通过邻接表给出。
* 算法要自己遍历邻接表，才能看到弧和弧长。
* 访问弧和做比较都要算时间。
* 这个模型更接近实际输入。

### universal optimality

* 它不是普通的 worst-case optimality，像是在固定一张图之后，看算法在这张图所有可能边权和表示方式下，能不能接近这张图本身的最优代价。
* 所以它比只看 `n, m` 的最坏情况更细。

## 4. 第 6 节：下界

第 6 节主要证明三种下界。

### 时间下界：`Ω(m)`

* 算法必须看足够多的输入。
* 如果有些邻接表或者弧没有看，就可能漏掉一条会改变距离顺序的弧。
* 所以在 time model 里，至少要有 `Ω(m)` 的时间。

### 比较下界：`Ω(log D)`

* `D` 是可能的 distance order 数量。
* 算法要区分这么多种可能顺序。
* 这和排序下界类似，决策树至少需要 `log D` 层。

### 比较下界：`Ω(F - n + 1)`

* `F` 是最多前向弧数。
* 如果前向弧很多，算法需要足够多比较来判断哪些弧会影响真实距离顺序。
* 这个下界和 Lemma 4.2 里的上界很对应。

## 5. 合并后的下界

* time model 下界：`Ω(m + log D)`。
* comparison model 下界：`Ω(F - n + 1 + log D)`。

后面如果 Dijkstra 或增强版 Dijkstra 能达到这些量级，就可以说明它是 universally optimal。

## 6. 目前的问题

* `D` 具体怎么估计还不太清楚。
* working-set heap 为什么能把 Dijkstra 做到 `O(m + log D)` 还需要继续看。
* `F - n + 1` 在 Lemma 4.2 和第 6 节下界里都出现，感觉它是连接算法上界和问题下界的关键。
