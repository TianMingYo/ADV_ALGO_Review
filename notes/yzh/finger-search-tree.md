# Finger Search Tree 笔记

## 1. 它解决什么问题

普通平衡二叉搜索树支持有序集合操作，比如查找、插入、删除，通常复杂度是：

* `O(log n)`

这里的 `n` 是整棵树里的元素个数。

Finger search tree 关注的是另一种情况：

* 我已经知道一个元素的位置；
* 我要在它附近查找或插入另一个元素；
* 新位置和已知位置之间只隔了 `k` 个元素。

这时 finger search tree 可以把代价做到：

* `O(log k)`，或者更准确地写成 `O(1 + log k)`。

这里的 `k` 不是总元素数，而是新位置离已知位置的“秩距离”。如果目标就在附近，操作就很快。

## 2. finger 是什么意思

这里的 finger 可以理解为“手指指着的位置”，也就是一个已经知道的元素指针。

比如有一个有序列表：

```text
1, 3, 5, 8, 11, 15, 20
```

如果我已经知道元素 `8` 的位置，现在要插入 `10`，它离 `8` 很近。普通平衡树还是从根开始找，代价是 `O(log n)`；finger search tree 可以从 `8` 附近开始找，代价更像 `O(log k)`。

如果要插入的元素就在相邻位置，`k` 很小，代价接近常数。

## 3. 和普通平衡树的区别

普通平衡树的思路是：

* 每次都从根开始；
* 不管目标离某个已知位置近不近；
* 代价按整棵树规模 `n` 算。

Finger search tree 的思路是：

* 从一个已知位置，也就是 finger，开始；
* 如果目标就在附近，就不用走很远；
* 代价按目标和 finger 的距离 `k` 算。

所以它适合处理“局部移动”或“局部插入”很多的场景。

## 4. 为什么第 9 节需要它

recursive Dijkstra 里要维护一个列表 `L`。

这个 `L` 表示当前构造出来的距离顺序。瓶颈点一开始已经按层数放进 `L`。当一个非 bottleneck 顶点 `v` 被扫描后，算法要把 `v` 插入到 `L` 的正确位置。

插入位置不是从头找，而是从它的父亲 `p(v)` 附近开始找：

* `p(v)` 是最短路树里 `v` 的父亲；
* `p(v)` 应该出现在 `v` 之前；
* `v` 的真实距离不小于 `p(v)` 的真实距离；
* 所以 `v` 的插入位置应该在 `p(v)` 后面某个地方。

因此，`p(v)` 就是一个天然的 finger。

如果从 `p(v)` 往后数 `k` 个位置才能找到 `v` 的插入点，那么 finger search tree 可以用：

* `O(1 + log k)`

完成这次插入。

## 5. 为什么这个局部代价重要

如果每次插入非 bottleneck 都用普通平衡树，代价可能是：

* 每次 `O(log n)`；
* 总共可能有很多次；
* 最后会得到一个太粗的上界。

但论文想要的比较次数是：

* `O(F - n + 1 + log D)`

其中多出来的维护列表 `L` 的代价，必须被压到 `O(log D)` 级别。

Finger search tree 给出的 `O(log k)` 很关键，因为它允许作者把每次插入的代价和“插入位置离父亲有多远”联系起来。这个距离可以进一步转成区间长度，然后用 Lemma 7.1 的区间 DAG 工具控制总和。

也就是说：

```text
finger search tree
  -> 每次插入代价是 log(局部距离)
  -> 局部距离可以表示成区间长度
  -> 区间长度的 log 和可以用 log D 控制
```

这就是 Lemma 9.3 的大概逻辑。

## 6. 和 working-set bound 的相似点

Finger search tree 和 working-set heap 的思想有点像。

working-set heap：

* 如果一个元素的 working set 小；
* 删除它就便宜；
* 代价不是只看全局规模。

finger search tree：

* 如果新位置离 finger 近；
* 查找或插入就便宜；
* 代价也不是只看全局规模。

它们都体现了 beyond-worst-case 的味道：不只用 `n` 这种全局最坏情况参数，而是利用操作序列或输入结构里的局部性。

## 7. 在第 9 节里的具体对应

在 Lemma 9.3 中：

* 顶点按扫描顺序记为 `v_1, v_2, ..., v_n`。
* 对每个顶点 `v_i`，设它的父亲 `p(v_i)` 是 `v_j`。
* 插入 `v_i` 时，从 `p(v_i)` 附近开始找。
* 论文把这个搜索范围看成一个区间。

如果 `v_i` 插入位置离父亲越远，这个区间就越长，finger search tree 的代价就越高。

但如果很多插入都离父亲很远，就说明列表中有很多可交换的顺序空间，也就意味着可能的 distance order 数量 `D` 很大。

因此，所有这些局部插入代价加起来，可以被 `O(log D)` 控制。

对这篇论文来说，该数据结构要点如下：

* 它维护一个有序列表。
* 它支持从已知元素附近开始查找和插入。
* 如果插入位置离 finger 的秩距离是 `k`，代价是 `O(1 + log k)`。
* 在 recursive Dijkstra 中，父亲 `p(v)` 就是插入 `v` 时的 finger。
* 它让维护最终距离顺序 `L` 的总代价可以被压到 `O(log D)`。
